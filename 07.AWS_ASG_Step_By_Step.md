# AWS Auto Scaling Group (ASG) - Complete Step-by-Step Guide

A comprehensive guide to creating and managing AWS Auto Scaling Groups using the AWS Management Console.

![Image](https://github.com/user-attachments/assets/25666d44-b6df-4f24-9a9f-25c70c7dfc8c)

## Prerequisites

Before creating an Auto Scaling Group, ensure you have:

- [ ] AWS Account with root or IAM user access
- [ ] EC2 Key Pair for SSH access
- [ ] VPC with public and private subnets (see [VPC Setup Guide](01.Aws_VPC.md))
- [ ] Security Group configured
- [ ] (Optional) Application Load Balancer already set up (see [ALB Guide](02.aws_ALB.md))

## Step 1: Prepare Your Custom AMI

Before creating an ASG, you need a custom AMI with your application pre-configured.

### Step 1.1: Launch a Base EC2 Instance

1. Go to **EC2 Dashboard** → **Launch Instance**
2. Configure:
   - **Name**: `template-instance`
   - **AMI**: Amazon Linux 2 (or your preferred AMI)
   - **Instance Type**: t2.micro
   - **Key Pair**: Select your key pair
   - **Network Settings**:
     - VPC: Select your custom VPC
     - Subnet: Public subnet
     - Auto-assign Public IP: Enable
     - Security Group: Select your web security group
   - **Storage**: 8 GiB gp3

### Step 1.2: Install Your Application

Connect to your instance and install the web server:

```bash
# Connect via SSH
ssh -i your-key.pem ec2-user@<public-ip>

# Update and install Apache
sudo yum update -y
sudo yum install httpd -y
sudo systemctl start httpd
sudo systemctl enable httpd

# Create your application content
echo "<h1>Auto Scaling Server</h1>" | sudo tee /var/www/html/index.html

# Test locally
curl localhost
```

### Step 1.3: Create AMI from Instance

1. Go to **EC2 Dashboard** → **Instances**
2. Select your instance (`template-instance`)
3. Click **Actions** → **Image and templates** → **Create image**
4. Configure:
   - **Image name**: `my-web-server-ami`
   - **Image description**: "Web server AMI for ASG"
   - **No reboot**: Checked (recommended)
5. Click **Create Image**
6. Note the **AMI ID** (e.g., ami-0abc123xyz)

> ⚠️ **Important**: Wait for the AMI status to become "Available" before proceeding.

## Step 2: Create Launch Template

The Launch Template defines the configuration for instances launched in the ASG.

### Step 2.1: Navigate to Launch Templates

1. Go to **EC2 Dashboard**
2. Scroll down to **Launch Templates** (under "Instances")
3. Click **Create launch template**

### Step 2.2: Configure Launch Template Basics

1. **Launch template name**: `my-asg-template`
2. **Version description**: "Version 1 - Web Server"
3. **Auto Scaling guidance**: Check "Provide guidance..."

### Step 2.3: Configure Amazon Machine Image (AMI)

- **AMI**: Select your custom AMI (`my-web-server-ami`)

### Step 2.4: Configure Instance Type

- **Instance Type**: t2.micro (or your preferred type)

### Step 2.5: Configure Key Pair

- **Key pair name**: Select your key pair

### Step 2.6: Configure Network Settings

1. Expand **Network Settings**
2. **VPC**: Select your custom VPC
3. **Security Groups**: Select your web security group
4. **Subnet**: Do NOT specify (will be specified in ASG)

### Step 2.7: Configure Storage

- **Volume (Root)**: Default 8 GiB gp3

### Step 2.8: Configure User Data (Bootstrap Script)

1. Expand **Advanced details**
2. Scroll to **User data**
3. Add the following script:

```bash
#!/bin/bash
# Bootstrap script for Auto Scaling Group

# Update packages
yum update -y

# Install Apache
yum install httpd -y

# Start and enable Apache
systemctl start httpd
systemctl enable httpd

# Create dynamic content
INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
AZ=$(curl -s http://169.254.169.254/latest/meta-data/availability-zone)

echo "<html>" > /var/www/html/index.html
echo "<head><title>ASG Server</title></head>" >> /var/www/html/index.html
echo "<body>" >> /var/www/html/index.html
echo "<h1>Auto Scaling Server</h1>" >> /var/www/html/index.html
echo "<p>Instance ID: $INSTANCE_ID</p>" >> /var/www/html/index.html
echo "<p>Availability Zone: $AZ</p>" >> /var/www/html/index.html
echo "<p>Server Time: $(date)</p>" >> /var/www/html/index.html
echo "</body>" >> /var/www/html/index.html
echo "</html>" >> /var/www/html/index.html
```

### Step 2.9: Create the Launch Template

1. Click **Create launch template**
2. Click **View launch templates** to confirm

## Step 3: Create Auto Scaling Group

### Step 3.1: Navigate to Auto Scaling Groups

1. Go to **EC2 Dashboard**
2. Find **Auto Scaling Groups** (under "Auto Scaling")
3. Click **Create Auto Scaling group**

### Step 3.2: Choose Launch Template

1. **Auto Scaling group name**: `my-asg`
2. **Launch template**: Select `my-asg-template`
3. **Launch template version**: Default (Latest)
4. Click **Next**

### Step 3.3: Configure Network

1. **VPC**: Select your custom VPC
2. **Availability Zones and subnets**: Select all private subnets
   - `private-subnet-1` (ap-south-1a)
   - `private-subnet-2` (ap-south-1b)
   - `private-subnet-3` (ap-south-1c)
3. Click **Next**

### Step 3.4: Configure Load Balancing (Optional but Recommended)

1. **Choose how to load balance**: Select "Attach to an existing load balancer"
2. **Attach to an existing target group**: Select your ALB target group
   - Target group: `my-alb-tg`
3. **Health checks**:
   - **Health check type**: ELB
   - **Health check grace period**: 300 seconds
4. Click **Next**

### Step 3.5: Configure Group Size

1. **Desired capacity**: 2
2. **Minimum capacity**: 1
3. **Maximum capacity**: 4
4. Click **Next**

> 💡 **Tip**: Start small and adjust based on your traffic needs.

### Step 3.6: Configure Scaling Policies (Optional)

Choose one of the following options:

#### Option A: No Scaling Policies (Manual)
- Select "None" if you want to manually adjust capacity

#### Option B: Target Tracking Scaling (Recommended)
1. Select "Target tracking scaling policy"
2. Configure:
   - **Metric type**: Average CPU utilization
   - **Target value**: 70%
3. Click **Add policy**

#### Option C: Step Scaling
1. Select "Step scaling"
2. Create CloudWatch alarm first

Click **Next**

### Step 3.7: Add Notifications (Optional)

1. Click **Add notification**
2. **Topic**: Create new topic
3. **Topic name**: `asg-notifications`
4. **Events**: Select events to notify
   - instance-launch
   - instance-terminate
   - instance-launch-failure
5. **Email**: Enter your email
6. Click **Next**

### Step 3.8: Add Tags (Recommended)

1. Click **Add tag**
2. Configure:
   - **Key**: Name
   - **Value**: ASG-Instance
   - **Propagate**: Checked
3. Click **Next**

### Step 3.9: Review and Create

1. Review all settings
2. Click **Create Auto Scaling group**

## Step 4: Verify Your ASG

### Step 4.1: Check ASG Status

1. Go to **Auto Scaling Groups**
2. Select `my-asg`
3. Check the **Details** tab:
   - Desired: 2
   - Min: 1
   - Max: 4
   - Status: Active

### Step 4.2: Check Instances Tab

1. Click the **Instances** tab
2. Verify:
   - Two instances are launching
   - Lifecycle state: "InService" (after a few minutes)
   - Health status: "Healthy"

### Step 4.3: Test via Load Balancer

1. Copy your ALB DNS name
2. Open in browser multiple times
3. Observe different Instance IDs in responses

## Step 5: Test Auto Scaling

### Test 5.1: Manual Scale-Out

1. Select your ASG
2. Click **Edit**
3. Change **Desired capacity** from 2 to 3
4. Click **Save**
5. Wait 2-3 minutes
6. Check **Instances** tab - new instance should launch

### Test 5.2: Manual Scale-In

1. Edit ASG again
2. Change **Desired capacity** from 3 to 2
3. Click **Save**
8. One instance will terminate

### Test 5.3: Automated Scaling (CPU Based)

If you configured target tracking:

1. Connect to one instance via SSH
2. Run a CPU stress test:

```bash
# Install stress tool
sudo yum install stress -y

# Stress the CPU
stress --cpu 80 --timeout 300 &
```

3. Watch the ASG - a new instance should launch when CPU > 70%
4. Stop the stress test - instance should terminate

## Step 6: Configure Auto Scaling Policies (Deep Dive)

### Step 6.1: Target Tracking Policy

1. Select ASG → **Automatic Scaling** tab
2. Click **Add policy**
3. Configure:
   - **Policy name**: cpu-scaling-policy
   - **Metric type**: Average CPU utilization
   - **Target value**: 70%
   - **Instances need**: 60 seconds warm-up
4. Click **Add policy**

### Step 6.2: Scheduled Scaling

1. Click **Add policy**
2. Configure:
   - **Policy name**: morning-scaleup
   - **Scaling schedule**: Recurring
   - **Recurrence**: cron(0 9 * * ? *)
   - **Action**: Set Desired Capacity to 4
3. Click **Add policy**

### Step 6.3: Step Scaling

1. Create CloudWatch Alarm first:
   - Go to CloudWatch → Alarms → Create Alarm
   - Metric: EC2 → Per-Instance Metrics → CPUUtilization
   - Threshold: > 80%
   - Action: Create new topic

2. Add Step Policy to ASG:
   - **Policy type**: Step scaling
   - **Metric**: Average CPU utilization
   - **Breaches**: > 80%
   - **Add step**:
     - Add 1 instance when breach > 80%
     - Add 2 instances when breach > 90%

## Step 7: Monitoring and Troubleshooting

### Step 7.1: View CloudWatch Metrics

1. Go to **Auto Scaling Groups** → Select your ASG
2. **Monitoring** tab
3. View:
   - CPU Utilization
   - Network In/Out
   - Request Count per instance

### Step 7.2: Common Issues

#### Issue: Instances Not Passing Health Checks

**Solution:**
1. Check security group allows traffic from ALB
2. Verify health check path returns 200 OK
3. Check user data script is working

#### Issue: ASG Not Scaling

**Solution:**
1. Verify scaling policies are configured
2. Check CloudWatch alarms are active
3. Verify minimum capacity allows scaling

#### Issue: Instance Launch Failures

**Solution:**
1. Check instance limits in AWS account
2. Verify key pair exists
3. Check VPC has available IPs
4. Review launch template for errors

## Step 8: Best Practices

### Security
- [ ] Always use custom AMIs with latest patches
- [ ] Use IAM roles instead of embedding credentials
- [ ] Restrict security groups to minimum required
- [ ] Enable termination protection

### Scaling
- [ ] Start with conservative min/max values
- [ ] Use multiple Availability Zones
- [ ] Configure appropriate health check grace period
- [ ] Test scaling policies before production

### Cost Optimization
- [ ] Use Reserved Instances for baseline capacity
- [ ] Use Spot Instances for fault-tolerant workloads
- [ ] Set appropriate CloudWatch alarm periods

### High Availability
- [ ] Deploy across 3+ Availability Zones
- [ ] Use ELB health checks
- [ ] Configure appropriate cooldown periods

## Step 9: Cleanup (Important!)

Follow these steps to avoid ongoing charges:

### Step 9.1: Delete Auto Scaling Group

1. Go to **Auto Scaling Groups**
2. Select `my-asg`
3. Click **Delete**
4. Confirm deletion

### Step 9.2: Delete Launch Template

1. Go to **Launch Templates**
2. Select `my-asg-template`
3. Click **Actions** → **Delete**

### Step 9.3: Delete Target Groups

1. Go to **Target Groups**
2. Select `my-alb-tg`
3. Click **Delete**

### Step 9.4: Delete Load Balancer

1. Go to **Load Balancers**
2. Select `my-alb`
3. Click **Actions** → **Delete**

### Step 9.5: Terminate Remaining Instances

1. Go to **EC2 Instances**
2. Terminate any running instances

### Step 9.6: Delete AMI and Snapshot

1. Go to **AMIs**
2. Select your AMI → **Actions** → **Deregister**
3. Go to **Snapshots**
4. Delete associated snapshot

## Quick Reference Commands

### AWS CLI - ASG Commands

```bash
# Create ASG
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name my-asg \
  --launch-template LaunchTemplateId=lt-xxx \
  --min-size 1 \
  --max-size 4 \
  --desired-capacity 2 \
  --vpc-zone-identifier "subnet-1,subnet-2"

# Update ASG
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name my-asg \
  --desired-capacity 3

# Delete ASG
aws autoscaling delete-auto-scaling-group \
  --auto-scaling-group-name my-asg

# Describe ASG
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names my-asg
```
