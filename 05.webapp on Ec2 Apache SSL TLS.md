# Securing Web App on AWS EC2 with Apache SSL/TLS

![Image](https://github.com/user-attachments/assets/2ed30ec6-a913-4310-b1de-f156033549a2)


## Project Concept
This project focuses on securing a web application hosted on an AWS EC2 instance using Apache by implementing SSL/TLS encryption. The goal is to protect data in transit between the client (browser) and server, preventing interception by hackers. Additionally, enabling HTTPS improves SEO (Search Engine Optimization) as search engines prioritize secure sites.

Key benefits:
- **Data Privacy**: Encrypts sensitive user data.
- **Application Integrity**: Prevents tampering or unauthorized access.
- **Secure Communication**: Uses port 443 for HTTPS instead of insecure HTTP on port 80.
- **Multi-Level Security**: Covers application, infrastructure, and network levels.

## Prerequisites
- AWS account with an running EC2 instance (e.g., Amazon Linux).
- SSH access to the EC2 instance (using .pem key).
- Apache (httpd) installed and serving a basic website on HTTP.
- Domain name (optional, but used in examples like `gurus.com`).


## Step-by-Step Guide (Exactly as in the Video, Starting from Step 2)

### Step 2: Understand the Need for Application Security
- Explain the importance of securing the web app:
  - Protect user data privacy.
  - Ensure productive support and reliable service.
  - Maintain application integrity against threats.
- Security layers discussed:
  - **Application Level**: Code and app-specific protections.
  - **Infrastructure Level**: Server and OS hardening.
  - **Network Level**: Control incoming/outgoing traffic.
- Emphasis: Without security, data can be intercepted easily on HTTP.

### Step 3: Implement SSL/TLS for Secure Communication
- Objective: Encrypt traffic using HTTPS on port 443.
- Port details:
  - HTTP: Port 80 (insecure).
  - HTTPS: Port 443 (secure with SSL/TLS).

#### Sub-Steps (Demonstrated in Video):
1. **Access AWS EC2 Console**:
   - Log in to AWS Management Console.
   - Navigate to EC2 > Instances.
   - Select your running instance with Apache installed.
   - Verify security group is attached.

2. **Configure Security Group for HTTPS**:
   - In EC2 console, go to the instance's **Security** tab > Click on the Security Group.
   - Edit **Inbound rules**.
   - Add a new rule:
     - Type: HTTPS
     - Protocol: TCP
     - Port range: 443
     - Source: 0.0.0.0/0 (allow from anywhere).
   - Save rules.
   - (Note: Ensure port 80 is already allowed for HTTP if needed.)

3. **Install and Configure SSL on Apache**:
   - SSH into the EC2 instance:
     ```bash
     ssh -i your-key.pem ec2-user@your-ec2-public-ip
     ```
   - Update system and install OpenSSL and mod_ssl:
     ```bash
     sudo yum update -y
     sudo yum install openssl mod_ssl -y
     ```

4. **Generate Private Key and Certificate Signing Request (CSR)**:
   - Generate RSA private key:
     ```bash
     openssl genrsa -out server.key 2048
     ```
   - Generate CSR:
     ```bash
     openssl req -new -key server.key -out server.csr
     ```
   - Fill prompts:
     - Country Name: (e.g., IN)
     - State/Province: (e.g., Telangana)
     - Locality: (e.g., Hyderabad)
     - Organization: (e.g., LWP Labs)
     - Organizational Unit: (e.g., DevOps)
     - Common Name (CN): Your domain (e.g., lwplabs.com)
     - Email: (e.g., info@lwplabs.com)
     - Skip challenge password and optional company name.

5. **Submit CSR to a Certificate Authority (CA)**:
   - Copy the CSR content (from `server.csr`).
   - Send to a trusted CA (e.g., DigiCert, GeoTrust, Let's Encrypt, or GoDaddy).
   - CA verifies and issues:
     - `certificate.crt` (signed certificate).
     - `CA_bundle.crt` (intermediate/chain certificates).
   - (Note: Self-signed certs are possible but not trusted by browsers; video recommends paid/trusted certs.)

6. **Configure Apache for SSL**:
   - Move files to appropriate locations (e.g., /etc/httpd/ssl/ or as per your setup).
   - Edit SSL config file:
     ```bash
     sudo vi /etc/httpd/conf.d/ssl.conf
     ```
   - Update directives:
     ```apache
     SSLEngine on
     SSLCertificateFile /path/to/certificate.crt  # e.g., /etc/httpd/ssl/certificate.crt
     SSLCertificateKeyFile /path/to/server.key    # e.g., /etc/httpd/ssl/server.key
     SSLCertificateChainFile /path/to/CA_bundle.crt  # e.g., /etc/httpd/ssl/CA_bundle.crt
     Listen 443
     <VirtualHost *:443>
         ServerName your-domain.com  # e.g., lwplabs.com
         DocumentRoot /var/www/html
         # Other settings as needed
     </VirtualHost>
     ```
   - Save and exit.

7. **Enable and Restart Apache Service**:
   - Enable SSL module (if needed):
     ```bash
     sudo a2enmod ssl
     ```
   - Restart Apache:
     ```bash
     sudo systemctl restart httpd
     ```
     OR
     ```bash
     sudo service httpd restart
     ```

8. **Verify SSL Configuration**:
   - Check port 443 listening:
     ```bash
     netstat -tuln | grep 443
     ```
   - Test in browser: Visit `https://your-ec2-public-ip` or `https://your-domain.com`.
   - Look for secure padlock icon.

### Step 4: Validate Security Group and Firewall Settings
- Double-check EC2 security group: Port 443 open.
- Optional: Allow port 80 for HTTP-to-HTTPS redirect.
- Add redirect in Apache config (in httpd.conf or .htaccess):
  ```apache
  RewriteEngine On
  RewriteCond %{HTTPS} off
  RewriteRule ^(.*)$ https://%{HTTP_HOST}$1 [R=301,L]
  ```

### Step 5: Use Trusted SSL Certificates (Recommended)
- Avoid self-signed certs (browsers show warnings).
- Options:
  - Free: Let's Encrypt (use Certbot for auto-install).
  - Paid: DigiCert (~₹499/year), GeoTrust, Sectigo.
- Cert details include: RSA encryption, serial number, issuer (e.g., DigiCert Global Root CA).

### Step 6: Test and Verify Security
- Access site via HTTPS.
- In browser: Click padlock > View Certificate.
- Verify:
  - Issued to: Correct domain.
  - Valid dates.
  - Issuer: Trusted CA (e.g., DigiCert, GoDaddy).
- Test for security: No mixed content warnings; all resources load over HTTPS.

## Additional Notes
- **SEO Benefit**: HTTPS sites rank higher in search results.
- **Cost**: SSL certs can be expensive for enterprises but essential.
- **Internal vs External**: No SSL needed for internal VPC traffic, but required for public access.
- **Practice Tip**: Follow along with your own EC2 instance for hands-on learning.
- For production: Use domain with Route 53, auto-renew certs, and monitor with AWS Certificate Manager (ACM) if integrating with ELB.
